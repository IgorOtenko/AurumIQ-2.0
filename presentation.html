The problem

Stock research is fragmented and slow.

A retail investor pulls from 5+ disconnected sources, manually synthesizes the picture, and re-does it every quarter per holding.

Today's workflow
  • Yahoo Finance for prices and earnings dates
  • Seeking Alpha for analyst sentiment
  • SEC 10-Qs for segment breakdowns
  • Twitter / Substack for catalysts
  • A spreadsheet to glue it together

Time per holding: 2–4 hours. Multiply by 10 holdings, every quarter.

What AurumIQ delivers
  • One view per ticker10 sections — numbers, trends, analyst setup, AI bull/bear thesis, sources, and analysis history
  • AI synthesis I can trustEvery figure traces to source data — invented numbers get rejected before they reach the UI
  • AutomatedScheduled daily regeneration + price/earnings alerts
The product

One dashboard. Ten sections. Per stock.

Five data sections, three AI sections, two depth sections — each lazy-mounted, each in its own error boundary so one failure can't crash the dashboard.

The critical engineering

AI synthesis with a three-layer validator.

messages.parse() + zodOutputFormat() enforce schema server-side. A traceability validator catches invented numbers but lets through legitimate derivations like implied upside.

  1. Structured output (server-side) Anthropic enforces a Zod-derived JSON schema; the model can't return free-form prose where JSON should be.
  2. Zod parse (client-side) SDK strips unsupported constraints (array lengths) and validates them in Zod after the response lands.
  3. Traceability check Every currency, percent, or large bare number in the output must match a source value — directly, by unit conversion, by compact scaling, or as a pairwise derivation.
On failure

Status → validation_failed. Prior content is left untouched — user keeps the last validated version with an amber warning. We never overwrite verified analysis with hallucinated numbers.

Real regression — NVDA verification

Output contained "36.6% upside". Substring search couldn't find 36.6 in the source.

Source data passed to the prompt
targetMeanPrice: 294.22,
currentPrice: 215.33
The validator's derivation check
// for every pair (a, b) in source
const ratio = (b - a) / a;
if (nearMatch(target, ratio * 100)) return true;

// (294.22 - 215.33) / 215.33 * 100 = 36.6 ✓

8 Vitest cases lock this regression behavior in.

Skill layer 1 · Superpowers

The protocols that made parallelism safe.

Process-level skills — they encode how to think about planning, execution, verification, and parallel work. Reached for before any code is written.

superpowers:writing-plans
Translated each phase spec into an actionable plan before any code. The 0X-PLAN.md files in .planning/phases/ are the artifact — frontmatter declares files_modified, must_haves.truths, and acceptance criteria.
superpowers:executing-plans
Ran plans phase by phase with atomic commits. Acceptance criteria explicitly checked before each phase closes — no "I think it works."
superpowers:subagent-driven-development
Decomposed phase work into sub-agent tasks with strict file ownership. Each sub-agent prompt declares its file scope so parallel writes never collide.
superpowers:dispatching-parallel-agents
5 concurrent agents per parallel wave. Phase 2's 6-adapter wave finished in one round-trip instead of six sequential ones.
superpowers:test-driven-development
44 unit + integration tests. Phase 2 wrote 22 tests alongside the adapters; Phase 7 added 8 validator regression tests for the AAPL + NVDA cases.
superpowers:verification-before-completion
The wiring gate. Every wave ends with tsc + lint + vitest + build in the main thread before commit. Catches anything the parallel agents couldn't see from inside their file slice.
superpowers:requesting-code-review
Peer-review loop on every PR. Catches subtle issues like the Pino worker-exit during HMR (Phase 3) that a single-agent flow would have missed.

Skill layer 3 · claude-mem

Cross-session memory so the project survives every /clear.

A multi-day project across multiple machines — without persistent memory, Claude forgets every architectural decision the moment a session ends.

claude-mem:learn-codebase
Primed the entire codebase into memory at session start. Eliminates the "what's in this repo" tax.
claude-mem:pathfinder
Mapped the codebase into feature groups (auth · finance · portfolio · ai · cron) so navigation stayed structural, not file-list-grep.
claude-mem:mem-search
Searched persistent memory at the start of every new session — "what did we decide about Decimal handling?" returns the actual decision, not a guess.
claude-mem:make-plan
Detailed phased plans informed by memory of past decisions — avoids re-litigating settled tradeoffs.
claude-mem:timeline-report
Generated journey reports — what shipped when, what blocked it, what got deferred.
Why this mattered

Without claude-mem, the AAPL 16.6% validator regression and the NVDA 36.6% derivation case would have been independent discoveries — solved twice, possibly differently. With memory, the second one resolved in one validator extension instead of two.

Cross-session continuity in practice
  • Phase 5 referenced Phase 2's caching strategy without re-reading the code
  • Phase 7 docs cited tradeoffs locked in Phase 3
  • Validator improvements compounded across phases
Skill layer 4 · claude-api

The Anthropic SDK domain expert.

Reached for whenever the work touched the Anthropic SDK directly — most heavily in Phase 5 (AI pipeline). Returns current patterns, not training-cutoff guesses.

What it surfaced for Phase 5
  • Model migration tableMapped retired claude-3-5-haiku → claude-haiku-4-5 and claude-3-7-sonnet → claude-sonnet-4-6.
  • messages.parse() + zodOutputFormat()The structured-output pattern that replaced the original "ask the model to return JSON" prompt engineering.
  • Adaptive thinkingNo more budget_tokens tuning — model decides depth per request. Effort control via output_config.effort.
  • Prompt caching mechanicsSonnet 4.6's 2048-token cache floor → why SYSTEM_PROMPT is intentionally verbose.
  • Per-language SDK patternsTypeScript SDK requires Zod v4 via the zod/v4 subpath — saved hours of "why does zodOutputFormat reject my schema."
// what the skill produced — the exact pipeline
await anthropic.messages.parse({
  model: 'claude-sonnet-4-6',
  max_tokens: 4096,
  system: [{
    type: 'text',
    text: SYSTEM_PROMPT,
    cache_control: { type: 'ephemeral' }
  }],
  messages: builder(sources),
  thinking: { type: 'adaptive' },
  output_config: {
    effort: 'high',
    format: zodOutputFormat(schema),
  },
});
The judgment

Sonnet 4.6 for on-demand quality. Haiku 4.5 for nightly scheduled runs. Same pipeline, one model parameter, same validator. Cost optimization without code duplication.

Skill layer 5 · frontend-design

Distinctive UI without falling into AI defaults.

Modern AI models default to cream backgrounds, Inter font, generic purple gradients. frontend-design is the antidote — it produces context-specific design with character.

What this skill enforced
  • Token-first designTailwind v4 CSS variables for every color. No magic hex anywhere in components.
  • Semantic color codingEmerald = positive, rose = negative, amber = warning, sky = informational, violet = AI. Used consistently across all 10 sections.
  • Cohesive component grammarCard with accent-left, pill with intent tones, SectionWrapper with error boundary. Same primitives compose every section.
  • Anti-slop checklistNo purple gradients, no Inter-on-white, no generic emoji decoration. Dark financial theme is the brief, not a default.

Round out the kit · MCPs · custom skills

Knowing what not to use is also engineering.

The skill registry is a tool inventory. Discipline is reaching for the right one — and refusing the ones that don't fit, no matter how trendy.

Used selectively

Token-window savings telemetry — checked at phase boundaries.

Built-in tools (Claude Code)
  • TodoWrite — phase + wave task tracking
  • Bash run_in_background + Monitor — long-running tasks (dev server, Docker)
  • ToolSearch — lazy tool loading; context stays for work, not catalogue
MCPs — available, deliberately not used

Figma · Gmail · Google Drive · Google Calendar — all authenticated and available. None used.

  • No design handoff (Tailwind tokens drive the UI directly)
  • No email integration in dev (Pino-to-console covers it)
  • No external doc storage (.planning/ + docs/ in repo)
  • No calendar coupling (cron jobs run in-process)

Adding MCP servers without need = context bloat + auth surface area for zero product value.

The principle

Every skill / tool / MCP in this project has a named justification. If I can't say "I used X for Y," it's not in the stack. That's the difference between an AI workflow and AI noise.
