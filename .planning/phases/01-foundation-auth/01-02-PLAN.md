<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> /api/auth/signup | Untrusted user input (email, password) crosses to server |
| browser -> /api/auth/[...nextauth] | Credentials cross to server for authentication |
| JWT cookie -> middleware | Session token validated on every protected route request |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | /api/auth/signup | mitigate | Zod validates email format and password min length 8; bcryptjs.hash with cost 12 |
| T-01-02 | Spoofing | Credentials authorize | mitigate | bcryptjs.compare for timing-safe comparison; return null (not error detail) on failure |
| T-01-03 | Information Disclosure | /api/auth/signup | mitigate | 409 on duplicate email reveals email exists -- acceptable tradeoff for UX; never expose stack traces |
| T-01-04 | Information Disclosure | login error | mitigate | Generic "Invalid email or password" message; never reveal which field is wrong |
| T-01-05 | Tampering | JWT session | mitigate | httpOnly cookie (Auth.js default); NEXTAUTH_SECRET signs JWT; no localStorage |
| T-01-06 | Elevation of Privilege | /dashboard routes | mitigate | middleware.ts matcher on /dashboard/:path* checks auth before rendering; server-side auth() check in layout as defense-in-depth |
| T-01-SC | Tampering | npm installs | mitigate | shadcn/ui components are copy-paste (no runtime dependency); next-auth@beta is official Auth.js package |
</threat_model>
